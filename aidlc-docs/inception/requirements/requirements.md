# 要件書 (Requirements) — 旅スポ（TABISPO）

旅先で、現在地周辺のおすすめ観光地・グルメをすぐに調べられるWebアプリ。

---

## 1. インテント分析サマリー (Intent Analysis)

| 項目 | 内容 |
|---|---|
| **ユーザーの要望** | 旅先で、おすすめの観光地やグルメをすぐ調べたい。起動時に現在地を取得し、その土地の観光地・おいしいお店を自動で調べ、ジャンル別にリスト表示する |
| **リクエスト種別** | 新規プロジェクト（Greenfield）／新規ユーザー向けアプリ |

| **複雑度見積もり** | 中程度（Moderate）。外部API連携と状態管理はあるが、MVPの機能範囲は限定的 |
| **アプリ名** | **旅スポ**（読み: たびスポ。旅＋スポット。英字表記は TABISPO。確定） |

## 3. 機能要件 (Functional Requirements)

### FR-1 現在地の取得
- 起動時にブラウザの位置情報（Geolocation）から現在地を取得する。
- 取得には事前にユーザーの許可が必要。許可された場合、その座標を以降の検索の基点とする。

### FR-2 位置情報が使えない場合のフォールバック
- ユーザーが位置情報を拒否、または取得に失敗した場合、**地名・住所をテキスト入力**して検索の基点を指定できる。
- これによりアプリは位置情報なしでも利用可能とする。

### FR-3 起動時の自動検索
- 現在地（またはFR-2の入力地点）を基点に、**「観光」「グルメ」の両カテゴリを自動で検索**する。
- 初期表示は**観光タブ**。タブ切替でグルメ結果を即時に表示できる（再検索不要となるよう結果を保持）。

### FR-4 ジャンル別表示（大分類）
- 画面上部に**「観光」「グルメ」の大分類タブ**を設ける。
- 各タブで該当カテゴリの結果一覧を表示する。

### FR-5 細分類による絞り込み
- 各大分類タブ内に、**細分類の絞り込みボタン**を設ける。
- MVPで用意する代表的な細分類:
  - **グルメ**: カフェ / ラーメン / レストラン / 居酒屋
  - **観光**: 寺社 / 公園・自然 / 美術館・博物館 / 名所
- 細分類は将来的に追加できる構成とする。

### FR-6 一覧表示
- 各カテゴリで**約10件**を表示する。
- 並び順は**現在地から近い順（距離優先）**。

### FR-7 スポット情報の表示
- 各スポットについて以下を表示する:
  - 名前 / 写真 / 現在地からの距離
  - 評価（星）/ 口コミ件数
  - 営業時間・定休日（および営業状況）

### FR-8 営業中フィルタ
- 「**現在営業中のみ**」を表示するフィルタを切り替えられる。

### FR-9 検索範囲の切替
- 検索半径を**ユーザーが切り替えられる**（例: 約1km / 約3km / 約5〜10km）。
- 変更時は新しい半径で再検索する。

### FR-10 ナビ連携（外部）
- 各スポットから、**外部のGoogleマップ**（アプリ/Web）を開いて経路案内できる。
- アプリ内に地図画面は持たない。

### FR-11 お気に入り
- 各スポットを**★で端末内（localStorage想定）に保存**できる。ログイン不要。
- **お気に入り専用タブ／ボタン**から、保存したスポットの一覧を表示できる。
- 保存の解除（★の取り消し）ができる。

### FR-12 エラー／0件時の表示
- 検索結果が0件: 「近くに見つかりませんでした」等のメッセージを表示。
- 通信エラー: 「通信に失敗しました。再試行してください」等のメッセージ＋**再試行ボタン**を表示。

### FR-13 表示言語
- UI表記は**日本語**とする。

## 4. 非機能要件 (Non-Functional Requirements)

### NFR-1 プラットフォーム / 動作環境
- Webアプリ（ブラウザ動作）。**主にスマートフォンのブラウザ**での利用を想定し、レスポンシブな画面とする（前提: 主用途はモバイル）。
- 対象は世界中の位置（地域固定なし）。

### NFR-2 システム構成
- **フロントエンド中心**の構成。MVPではバックエンドサーバーを持たない。
- Google Places API のキーは、**Google Cloud側の利用制限**（HTTPリファラー制限・API種別制限・利用上限等）で保護する。
- 注意（前提共有）: フロントエンドのみの構成ではAPIキーがブラウザに露出しうるため、上記のGoogle側制限の設定を必須とする。将来、課金・不正利用が懸念される場合はバックエンド経由方式への移行を検討する。

### NFR-3 デプロイ / ホスティング
- MVPは**ローカル環境で動作**すればよい。インターネット公開は範囲外（将来検討）。

### NFR-4 コスト
- Google Places API は無料枠＋従量課金。起動時に2カテゴリを検索するため、呼び出し回数に留意する。
- 将来的な最適化余地: 結果のキャッシュ、検索タイミングの調整等（MVPでは必須としない）。

### NFR-5 ユーザビリティ
- 起動から一覧表示までを素早く、迷わず使えること。
- 約10件のコンパクトな一覧で軽快な閲覧性を確保する。

### NFR-6 セキュリティ / テスト方針
- セキュリティ拡張ルール: **無効**（MVP・学習/試作段階）。ただしNFR-2のAPIキー保護は実施する。
- プロパティベーステスト(PBT): **無効**。テストの詳細粒度はBuild & Test段階で決定する。

## 7. キーとなる要件サマリー (Key Summary)

- **コア体験**: 起動 → 現在地取得（拒否時は地名入力）→ 周辺の「観光」「グルメ」を自動検索 → 大分類タブ＋細分類絞り込みで、距離が近い順に約10件をリスト表示。各スポットは写真・評価・営業時間付きで、タップでGoogleマップのナビへ。
- **特徴**: ログイン不要ですぐ使える／お気に入りを端末内に保存／営業中フィルタ・検索範囲切替で旅先の「今すぐ」に応える。
- **MVPの割り切り**: 地図画面・多言語・公開はスコープ外。フロントエンド中心・ローカル動作で、まず「現在地→ジャンル別一覧」を確実に動かす。
